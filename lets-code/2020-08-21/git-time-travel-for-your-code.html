Git, time travel for your code

Scenario #1. Imagine that you are working on a very important report for school or a spreadsheet for work. It's getting late and you are making changes hastily and half asleep. When you review your work in the morning you realize that it wasn't your best work and you wish you could undo the chances you made the night before. But you can't because you have already closed and reopened your document, thus erasing your undo history.

Scenario #2. Imagine that you are working on a very important report for school or a spreadsheet for work. You finish it, hit save, and go to sleep delighted that you have finished it and have it all ready to be delivered the next day. However, when you wake up the next morning you find that your computer's harddrive died and your hard work is all gone.

Once upon a time programmers had the same issues, but after much lost work they realized there had to be a better way. So they created version control systems (aka source control systems). The most popular by far is called git. Git allows you to commit changes to your code as you go along in what they call a local code repository (or just repos for short), giving you a rich history of each change, and allowing you to go back and recover the state of your code at a specific point in time.

But that only solves the unlimited "undos" problem, not the harddrive dying. To solve that, git makes it easy to push your changes to a code repo that is online. The biggest provider of such online code repos is github.com.

Eventually, programmers made tools that add version control to other tools that you use every day. For example, did you know that Google Docs keeps a history of the changes that you make to your files?
